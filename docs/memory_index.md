# Memory Index

`harness-lab` needs more than isolated candidate folders. It needs a queryable summary layer that can answer:

- what failed recently
- which harness components are overrepresented
- which failure modes cluster together
- which candidates touched the same part of the harness

## First Memory Artifact

The first memory artifact is a filesystem-native candidate index:

```text
artifacts/memory/
  candidate_index.json
```

It is generated by scanning:

- `artifacts/candidates/*/workspace.json`
- `artifacts/candidates/*/proposal.json`
- `artifacts/candidates/*/diagnosis/summary.json`

## What The Index Contains

- per-candidate summary rows
- counts by proposal status
- counts by failure mode
- counts by harness component
- lineage summary for parent-child relationships

## Why This Matters

This is the bridge from:

- "many raw candidate workspaces"

to:

- "a memory layer that the proposer and decision system can query"

It is still intentionally simple. The point is to make cross-run synthesis possible before building a more complex judgment model.
